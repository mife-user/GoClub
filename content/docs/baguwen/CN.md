# 计算机网络
## 基础

- 输入一个URL会发生什么？
- 答案
    
    输入URL到页面展示的全过程： 1. DNS域名解析浏览器先查缓存，没有就递归查询DNS服务器，把域名解析成IP。2. TCP连接浏览器和服务器建立TCP三次握手；如果是HTTPS，还会进行TLS握手。3. 发送HTTP/HTTPS请求构建请求头、请求体，发送给服务器。4. 服务器处理并返回响应服务器处理请求，返回HTML、CSS、JS等资源。5. 浏览器渲染页面解析HTML构建DOM树，解析CSS构建CSSOM树，合成渲染树，布局绘制。6. 断开连接数据传输完毕，进行TCP四次挥手。
    
- 域名和IP的关系
- 答案
    
    IP 是服务器在网络中的唯一地址，由数字组成，不易记忆。域名是方便人类记忆的别名，通过 DNS 域名解析 映射到对应 IP。 - 一个 IP 可以对应多个域名（共享服务器、虚拟主机）。- 一个域名也可以对应多个 IP（DNS 负载均衡，提高可用性）。DNS 就是负责把域名翻译成 IP 的分布式服务。
    
- DNS协议
- 答案
    
    DNS 是应用层协议，作用是把域名解析成 IP 地址，基于UDP 53端口，是一个分布式、分层的域名系统。 解析流程（从右往左）： 1. 先查根域名服务器，找到顶级域（.com）服务器2. 再查**.com 顶级域名服务器**，找到对应二级域名服务器3. 最后查权威域名服务器，得到最终 IP 并返回   极简背诵版 DNS 是域名转IP的应用层协议，用UDP 53，分层、从右往左解析。
    
- 访问一个网站出现了问题，如何确定问题所在
- 答案
    
    访问网址出错，排查思路从上到下： 1. 看浏览器 HTTP 状态码- 4xx：客户端问题（权限、参数、路径）- 5xx：服务器内部错误2. ping 目标IP/域名- 通：网络层没问题，问题在应用层- 不通：网络不通或被禁 ping3. 用 tracert / traceroute 查链路看在哪一跳中断，定位是内网、运营商还是服务器端问题。4. 用 telnet 或 nc 测端口测试 80/443 端口是否开放，判断服务是否监听、防火墙是否拦截。5. 检查 DNSnslookup/dig 看域名是否解析正确。
    

## HTTP

- HTTP1/HTTP2/HTTP3区别
- 答案
    
    HTTP/1.1、HTTP/2、HTTP/3 核心区别：
    
    1. HTTP/1.1
    
    - 串行请求，队头阻塞
    - 纯文本，头部冗余
    - 性能较差。
    
    2. HTTP/2
    - 基于 TCP
    - 二进制分帧 + 多路复用，一个连接并行多个请求，解决应用层队头阻塞
    - HPACK 头部压缩
    - 仍受 TCP 队头阻塞 影响。
    
    3. HTTP/3
    - 基于 QUIC 协议 + UDP
    - 彻底解决 TCP 队头阻塞
    - 连接迁移更快、加密更安全
    - 可靠性在应用层保证。
- HTTPS的S是什么？讲一下原理
- 答案
    
    HTTPS 里的 S 是 Secure（安全），基于 HTTP + TLS/SSL 实现加密传输。 核心原理：混合加密 + 数字证书 1. 非对称加密：用来安全交换会话密钥，解决密钥传输问题。2. 对称加密：使用协商好的会话密钥加密实际数据，速度快。3. CA 数字证书：- 证书包含服务器公钥、域名、签名- 由权威机构签发，防止中间人攻击，确保公钥可信。 流程一句话：先通过 TLS 握手，用非对称加密安全协商会话密钥，之后用会话密钥做对称加密传输数据，既安全又高效。
    
- 为什么又HTTP还要有RPC，还要有WebSocket
- 答案
    
    HTTP、RPC、WebSocket 解决的是完全不同的场景问题，所以缺一不可： 1. HTTP- 通用、无状态、适合对外接口、浏览器访问- 简单通用，但开销大、不适合长连接、内部微服务不够高效2. RPC- 用于微服务内部调用- 像调用本地方法一样调用远程服务- 序列化更快（Protobuf）、性能更高、调用更简洁- 解决：内部服务通信效率低、接口不规范的问题3. WebSocket- 基于 TCP 的全双工长连接- 一次握手，永久连通- 解决：HTTP 不能服务端主动推送、频繁请求开销大- 用于聊天室、直播、实时通知等
